### Turso 事务缺陷（db/index-turso.js 第 240-260 行）

```javascript
const runInTransaction = async (callback) => {
  return enqueueWrite(async () => {
    const tx = {...};
    try {
      const result = await callback(tx);
      return result;
    } catch (err) {
      throw err;  // ❌ 抛出异常，但没有 rollback
    }
  });
};

// Turso 不支持 rollback
rollback: async () => {
  console.warn('[Turso] rollback() 被调用，但 Turso 不支持回滚');
  return;  // ❌ 空操作

## 四、问题原因总结

| 层级 | 问题 | 说明 |
|------|------|------|
| **业务层** | 上桌状态校验时机 | 校验在更新申请状态**之后**，而非之前 |
| **数据层** | Turso 不支持回滚 | 异常抛出后，申请状态无法回滚到待处理 |
| **系统层** | Timer 未设置 | 异常在设置 Timer 之前抛出，导致无计时器 |

### 问题流程图

```

    ↓

## 六、建议修复方案

### 短期方案

1. **调整校验顺序**: 将上桌状态校验移到更新申请状态**之前**
   ```javascript
   // 先检查上桌状态
   if (isOnTable) {
     throw { status: 400, error: '正在上桌，无法审批' };
   }
   // 再更新申请状态
   await tx.run('UPDATE applications SET status = ? ...');
   ```

2. **添加状态回滚**: 在异常捕获时手动恢复状态
   ```javascript
   try {
     // ...
   } catch (err) {
     // Turso 无 rollback，手动恢复
     await tx.run('UPDATE applications SET status = 0 WHERE id = ?', [id]);
     throw err;
   }
   ```

### 长期方案

3. **使用 Turso 事务替代方案**: 考虑在业务层面实现补偿机制
4. **添加数据一致性检查**: 定期检查 extra_data 与 status 是否匹配
