| 14:00~23:59 | 当前整点~23:00, 次日00:00, 次日01:00 | 今日h+ / 次日0-1 |

```javascript

### 4.1 后端 `lejuan-records.js` — 修复校验2

**修改前：**
```javascript
// 校验2: 小时必须在窗口内 (14~23 或 0~2)

**修改后：**
```javascript
// 校验2: 小时必须在窗口内 (14~23 或 0~1)

### 8.2 API 绕过防护

前端只负责 UI 展示，**后端是最终防线**：

- 即使前端不显示 hour=2 的选项，恶意请求直接调 API 传 hour=2 也会被后端拒绝（校验2）
- 传 hour=3~13 同样被后端拒绝
- 传过去时间被校验4拒绝
- 传日期不匹配被校验3拒绝

### 8.3 未来日期处理

当前实现允许选择未来日期（picker 无 `end` 限制）。选择未来日期时：
- 前端仍然按上述规则显示可选小时
- 后端校验3（日期匹配）会校验 `schedDate === 今日` 或 `=== 明日`
- **如果选择后天及以后的日期**，后端校验3会拒绝（因为日期不匹配今日/明日）

这是合理的：乐捐报备只允许预约今日14:00~次日01:00的时段。

### 8.4 非整点时间

后端校验1已处理：分钟≠00或秒≠00的请求会被拒绝，返回 `"预约时间必须是整点（分钟=00）"`。
