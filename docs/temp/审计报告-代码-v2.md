## 一、批量助教班次修改功能

### 1.1 PUT /api/coaches/batch-shift 是否实现

**状态：⚠️ 已实现但有严重缺陷**

**代码位置：** `/TG/tgservice/backend/routes/coaches.js` 第183-236行

**实现情况：**
- ✅ API 路由已注册：`app.use('/api/coaches/v2', coachesV2Router)`（server.js 第110行）
- ✅ 实际路径：`PUT /api/coaches/v2/batch-shift`
- ✅ 有班次枚举值验证（早班/晚班）
- ✅ 有事务包裹
- ✅ 有操作日志记录

**❌ 缺陷1：批量修改班次时，未联动更新水牌状态**

批量修改班次只更新了 `coaches` 表的 `shift` 字段，**没有联动更新 `water_boards` 表的状态**。

对比单个修改班次 API（PUT /api/coaches/:coach_no/shift），它正确地做了状态联动映射：
```javascript
// 单个修改有联动（routes/coaches.js 第286-323行）
const statusMap = {
  '早班空闲': '晚班空闲',
  '晚班空闲': '早班空闲',
  '早班上桌': '晚班上桌',
  '晚班上桌': '早班上桌',
  '早加班': '晚加班',
  '晚加班': '早加班'
};
```

但批量修改 API（第183-236行）**完全没有这段逻辑**。如果一个助教当前水牌状态是"早班空闲"，批量将其班次改为"晚班"后，水牌状态仍然是"早班空闲"——这会导致水牌显示与班次不一致。

**❌ 缺陷2：批量修改操作日志粒度不足**

批量修改的操作日志只记录了一条笼统的记录：
```javascript
old_value: JSON.stringify({ coach_no_list, shift: '原班次' }),
new_value: JSON.stringify({ coach_no_list, shift }),
remark: `批量修改${coach_no_list.length}名助教班次为${shift}`
```

没有记录每个助教原来的班次值，无法追溯每个助教的具体变更。

**修复建议：**
1. 在批量修改 API 的事务中，遍历每个助教，联动更新其水牌状态
2. 批量修改前查询每个助教的原始班次，在日志中记录详细变更

## 四、数据库 Schema 问题

### 4.1 coaches 表缺少 shift 列定义

**状态：❌ 严重问题**

代码位置：`/TG/tgservice/backend/init-db.js` 第105-120行

`coaches` 表的 CREATE TABLE 语句中**没有 `shift` 字段**：

```sql
CREATE TABLE IF NOT EXISTS coaches (
  coach_no TEXT PRIMARY KEY,
  employee_id TEXT,
  stage_name TEXT,
  real_name TEXT,
  id_card_last6 TEXT,
  level TEXT DEFAULT '初级',
  price REAL DEFAULT 2.3,
  age INTEGER,
  height INTEGER,
  photos TEXT,
  video TEXT,
  intro TEXT,
  is_popular INTEGER DEFAULT 0,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
)
```

所有班次相关代码（routes/coaches.js 中的 `SELECT ... shift FROM coaches`、`UPDATE coaches SET shift = ?`）都依赖这个字段。

**如果数据库中没有通过 ALTER TABLE 手动添加过 `shift` 列，所有班次相关功能将完全无法工作。**

**建议：** 在 `init-db.js` 中为 coaches 表添加 `shift` 列，或添加迁移脚本：
```sql
ALTER TABLE coaches ADD COLUMN shift TEXT DEFAULT '早班';
```
