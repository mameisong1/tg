## 问题清单

### 1. DDL 完整性问题

#### 1.1 admin_users 表 - 缺少权限字段索引
- **问题描述**: `role` 字段用于权限校验，但未创建索引。在频繁查询用户权限时会影响性能。
- **严重程度**: 低
- **建议**: 添加 `CREATE INDEX idx_admin_users_role ON admin_users(role);`

#### 1.2 coaches 表 - DDL 不一致
- **问题描述**: 设计方案中 `coaches` 表的修改说明与完整重建语句不一致。修改说明只提到 `ALTER TABLE coaches ADD COLUMN shift`，但完整重建语句中包含了所有字段。如果表已存在且需要添加字段，应明确说明 SQLite 的 ALTER TABLE 限制。
- **严重程度**: 中
- **建议**: 明确数据迁移策略，如需重建表，需提供数据迁移脚本（INSERT INTO new_table SELECT * FROM old_table）

#### 1.3 water_boards 表 - 缺少唯一约束
- **问题描述**: `coach_no` 应该有唯一约束，确保每个助教只有一个水牌记录。当前只设置了外键，没有唯一约束。
- **严重程度**: 高
- **建议**: 添加 `UNIQUE(coach_no)` 约束

#### 1.4 guest_invitation_results 表 - 缺少状态字段
- **问题描述**: 约客结果表只有 `result` 字段（未约客/约客有效/约客无效），但缺少审查状态字段（待审查/已审查）。无法区分"还未审查"和"审查后无效"。
- **严重程度**: 高
- **建议**: 新增 `review_status` 字段（0=待审查，1=已审查），或修改 `result` 枚举增加"待审查"状态

---

### 2. API 完整性问题

#### 2.1 水牌管理 API - 缺少批量操作接口
- **问题描述**: 只有单个助教的水牌状态更新接口 (`PUT /api/water-boards/:coach_no/status`)，没有批量更新接口。当需要批量设置水牌状态时（如集体下班），效率低下。
- **严重程度**: 中
- **建议**: 新增 `PUT /api/water-boards/batch-status` 接口

#### 2.2 服务单 API - 缺少删除接口
- **问题描述**: 服务单只有创建、查询、更新状态接口，没有删除接口。如果误创建服务单，无法删除。
- **严重程度**: 低
- **建议**: 新增 `DELETE /api/service-orders/:id` 接口（或软删除）

#### 2.3 上下桌单 API - 缺少删除接口
- **问题描述**: 同服务单，缺少删除接口。
- **严重程度**: 低
- **建议**: 新增 `DELETE /api/table-action-orders/:id` 接口

#### 2.4 申请事项 API - 缺少批量审批接口
- **问题描述**: 加班/公休申请只能逐个审批，没有批量审批接口。当有大量申请时，操作繁琐。
- **严重程度**: 低
- **建议**: 新增 `PUT /api/applications/batch-approve` 接口

#### 2.5 约客管理 API - 缺少查询接口
- **问题描述**: 约客记录只有提交、审查、统计接口，缺少查询接口。无法获取某个助教的约客历史记录，也无法获取某天的所有约客记录列表。
- **严重程度**: 高
- **建议**: 新增 `GET /api/guest-invitations` 接口，支持按日期、班次、助教筛选

#### 2.6 助教管理 API - 缺少查询接口
- **问题描述**: 只有批量修改班次、上班、下班接口，缺少获取助教列表、获取单个助教详情的接口（虽然可能有原有的助教管理 API，但设计方案中未提及）。
- **严重程度**: 中
- **建议**: 补充说明是否复用现有 API，或新增 `GET /api/coaches` 和 `GET /api/coaches/:coach_no`

#### 2.7 缺少操作日志 API
- **问题描述**: 注意事项中提到"所有状态变更需要记录操作日志"，但设计方案中没有操作日志的表结构和 API。
- **严重程度**: 高
- **建议**: 新增 `operation_logs` 表和对应的查询 API

---

### 3. 状态机逻辑问题

#### 3.1 状态转换矩阵 - 缺少"下班"状态的转换
- **问题描述**: 状态转换矩阵中，"下班"状态只有→上班按钮→早班空闲/晚班空闲的转换。但实际场景中，可能存在"下班"状态直接→休息/请假的情况（如明天请假今天已下班）。
- **严重程度**: 低
- **建议**: 确认业务规则，如下班后不能直接设置休息/请假，则保持现状；如可以，需补充转换规则

#### 3.2 状态转换矩阵 - "加班"状态的下班流程缺失
- **问题描述**: 状态转换矩阵中，"早加班/晚加班"状态只有→上班按钮→早班空闲/晚班空闲的转换。但加班状态是否可以直接下班？设计方案未明确。
- **严重程度**: 中
- **建议**: 补充加班状态的下班流程：早加班/晚加班 → 下班按钮 → 下班

#### 3.3 按钮可用性矩阵 - 逻辑不一致
- **问题描述**: 按钮可用性矩阵中，"早班空闲"状态的"下班按钮"为✅，但状态转换矩阵中只有"早班空闲→下班按钮→下班"。然而，如果助教在早班空闲状态想直接下班，是否需要先点击"下班按钮"？这与"上班按钮"的存在矛盾（早班空闲状态已经有上班按钮为✅，但此时助教已经在班）。
- **严重程度**: 中
- **建议**: 澄清"上班按钮"的含义：
  - 如果"上班按钮"是指"从非在班状态进入在班状态"，则早班空闲/晚班空闲状态的上班按钮应为❌
  - 如果"上班按钮"是指"从加班/休息/请假/乐捐状态恢复为空闲状态"，则早班空闲/晚班空闲状态的上班按钮应为❌

#### 3.4 状态枚举值 - 缺少"待审查"状态
- **问题描述**: 约客结果表的 `result` 字段枚举为"未约客/约客有效/约客无效"，但缺少"待审查"状态。助教上传约客记录后，在管理员审查前，状态应该是什么？
- **严重程度**: 高
- **建议**: 修改 `result` 枚举为：待审查/未约客/约客有效/约客无效

#### 3.5 乐捐报备 - 状态转换缺失
- **问题描述**: 状态转换矩阵中提到"任意状态→乐捐报备提交→乐捐（无需审批）"，但乐捐状态如何恢复到空闲状态？只有通过"上班按钮"吗？如果助教乐捐后直接下班呢？
- **严重程度**: 中
- **建议**: 补充乐捐状态的完整转换流程：
  - 乐捐 → 上班按钮 → 早班空闲/晚班空闲
  - 乐捐 → 下班按钮 → 下班（需确认业务规则是否允许）

---

### 4. 约客管理流程问题

#### 4.1 时间检查逻辑缺失
- **问题描述**: 设计方案提到"早班上传截止 16:00，晚班上传截止 20:00"，但 API 设计中没有限制提交时间的逻辑。助教是否可以在 16:00 后提交早班约客记录？
- **严重程度**: 高
- **建议**: 在 `POST /api/guest-invitations` 接口中添加时间校验：
  - 早班约客记录：只能在当天 16:00 前提交
  - 晚班约客记录：只能在当天 20:00 前提交
  - 或明确业务规则是否允许补交

#### 4.2 审查流程 - 缺少"驳回重传"机制
- **问题描述**: 约客审查只有"约客有效/约客无效"两种结果。如果审查为"约客无效"，助教是否可以重新上传？设计方案未明确。
- **严重程度**: 中
- **建议**: 明确审查为"无效"后的流程：
  - 方案 A: 不允许重新上传，记录为无效
  - 方案 B: 允许重新上传，覆盖原记录

#### 4.3 统计生成 - 缺少触发机制
- **问题描述**: `POST /api/guest-invitations/statistics` 接口是手动触发还是自动触发？如果是手动触发，谁在什么时候触发？如果是自动触发，定时任务设计缺失。
- **严重程度**: 中
- **建议**: 明确统计生成机制：
  - 方案 A: 管理员手动点击生成统计
  - 方案 B: 定时任务在审查完毕后自动生成（需补充定时任务设计）

#### 4.4 应约客定义 - 逻辑不完整
- **问题描述**: 应约客定义为"水牌状态为早班空闲/晚班空闲"，但以下情况未明确：
  - 助教在 16:00 时处于"早班上桌"状态，是否算应约客？
  - 助教在 16:00 时处于"早加班"状态，是否算应约客？
  - 助教在 16:00 后才上班（16:00 时状态为"下班"），是否算应约客？
- **严重程度**: 高
- **建议**: 明确应约客的完整定义：
  - 方案 A: 16:00 时水牌状态为"早班空闲"或"早班上桌"的助教
  - 方案 B: 当天班次为早班的所有助教（无论当前状态）
  - 方案 C: 16:00 前已经上班的所有早班助教

#### 4.5 约客记录与申请事项的重复
- **问题描述**: 设计方案中同时存在 `applications` 表（application_type 包含"约客记录"）和 `guest_invitation_results` 表。两者都存储约客相关数据，存在重复。
- **严重程度**: 高
- **建议**: 明确两者的区别和用途：
  - 方案 A: 删除 `applications` 表中的"约客记录"类型，统一使用 `guest_invitation_results` 表
  - 方案 B: 明确 `applications` 表的"约客记录"用于审批流程，`guest_invitation_results` 表用于统计

### 5. 其他问题

#### 5.1 权限矩阵 - 服务员权限过窄
- **问题描述**: 服务员角色的所有后台权限均为❌，但可以创建服务单（前台页面）。服务员是否能查看自己创建的服务单状态？
- **严重程度**: 低
- **建议**: 明确服务员是否可以查看服务单状态（新增"服务单查看"权限）

#### 5.2 OSS 配置 - 缺少 service_orders 图片目录
- **问题描述**: 服务单的 `requirement` 字段可能包含图片（如"这个桌子坏了"配图），但 OSS 配置中没有服务单图片目录。
- **严重程度**: 低
- **建议**: 如服务单支持图片，新增 `serviceOrderDir: 'service-orders/'` 配置

#### 5.3 定时任务 - 缺少具体实现说明
- **问题描述**: 定时任务只提到"每小时清理一次过期记录"，但没有说明使用什么工具（node-cron? 系统 crontab?），也没有说明清理脚本的路径。
- **严重程度**: 低
- **建议**: 补充定时任务的实现方式，如：
  ```javascript
  // backend/cron.js
  cron.schedule('0 * * * *', async () => {
    // 清理 2 小时前的已完成/已取消记录
  });
  ```

## 建议

### 高优先级（必须修改）

1. **water_boards 表添加唯一约束**: `UNIQUE(coach_no)`，防止一个助教有多个水牌
2. **guest_invitation_results 表增加审查状态**: 区分"待审查"和"已审查"
3. **新增约客记录查询 API**: `GET /api/guest-invitations`，支持筛选
4. **新增操作日志表**: 记录所有状态变更操作
5. **约客提交时间校验**: 在 API 中添加截止时间检查
6. **明确应约客定义**: 完善统计逻辑

### 中优先级（建议修改）

1. **coaches 表迁移策略**: 明确 SQLite ALTER TABLE 限制和数据迁移方案
2. **状态转换矩阵完善**: 补充加班状态的下班流程、澄清上班按钮含义
3. **约客审查驳回机制**: 明确无效后是否可重新上传
4. **统计生成机制**: 明确手动触发还是自动触发
5. **水牌批量操作 API**: 提升操作效率

### 低优先级（可选优化）

1. **admin_users 表 role 字段索引**: 性能优化
2. **服务单/上下桌单删除接口**: 完善 CRUD
3. **申请事项批量审批**: 提升操作效率
4. **助教查询 API**: 补充说明是否复用现有 API
5. **服务员权限扩展**: 查看服务单状态
6. **定时任务实现细节**: 补充技术实现

---

## 审计总结

设计方案整体框架完整，覆盖了数据库、API、状态机、权限、前端等核心模块。但存在以下关键问题需要修订：

1. **数据完整性**: 部分表缺少关键约束（唯一约束、状态字段）
2. **API 覆盖度**: 缺少查询接口、删除接口、日志接口
3. **状态机严谨性**: 部分状态转换逻辑不清晰，按钮可用性矩阵存在矛盾
4. **约客流程**: 时间检查、审查流程、统计逻辑需要完善

建议先修订高优先级问题，再进行开发实施。
