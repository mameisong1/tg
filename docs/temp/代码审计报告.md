## 问题清单

### 🔴 严重问题（必须修订）

#### 1. 约客记录提交逻辑与设计方案不一致

**问题描述**: 
- 设计方案 2.7.1 节规定：约客记录由**前端直接上传 OSS**，后端 API 只接收 `invitation_image_url`
- 但 `applications.js` 中 `POST /api/applications` 接口仍然处理"约客记录"类型申请（第 34-42 行）
- 这导致约客记录有两条提交路径：`/api/applications` 和 `/api/guest-invitations`，逻辑混乱

**位置**: 
- `/TG/tgservice/backend/routes/applications.js:34-42`
- `/TG/tgservice/backend/routes/applications.js:52-56`（约客记录验证逻辑）

**严重程度**: 🔴 高

**影响**: 
- 约客记录可能被提交到错误的表（applications 而非 guest_invitation_results）
- 约客统计功能无法正确获取数据

---

#### 2. 乐捐报备自动更新水牌状态的逻辑存在 SQL 注入风险

**问题描述**: 
- `applications.js` 第 59-61 行使用字符串拼接构建 SQL 查询：
```javascript
WHERE date(a.created_at) >= date('now', '-${parseInt(days, 10)} days')
```
- 虽然使用了 `parseInt` 进行转换，但仍属于不安全的 SQL 构建方式

**位置**: `/TG/tgservice/backend/routes/applications.js:59-61`

**严重程度**: 🔴 高

**修复建议**: 使用参数化查询：
```javascript
WHERE date(a.created_at) >= date('now', ?)
```
并传入参数 `['-' + days + ' days']`

---

#### 3. 加班审批通过后未处理水牌状态转换的边界情况

**问题描述**: 
- `applications.js` 第 214-245 行：加班审批通过后自动更新水牌状态
- 但代码未检查当前水牌状态是否允许转换（例如：如果助教已经是"早班上桌"状态，直接覆盖为"早加班"会导致状态不一致）
- 设计方案 3.2 节明确规定状态转换需要遵循转换矩阵

**位置**: `/TG/tgservice/backend/routes/applications.js:214-245`

**严重程度**: 🔴 高

**影响**: 
- 可能导致水牌状态违反状态转换规则
- 例如：助教正在上桌服务时被审批为加班，状态直接变为"早加班"，但实际仍在台上服务

---

#### 4. 约客统计生成逻辑缺少"所有记录审查完毕"校验

**问题描述**: 
- 设计方案 2.7.3 节和 7.1 节规定：约客统计生成条件为"16:00/20:00 后 + 所有早班/晚班助教约客记录审查完毕"
- 但 `guest-invitations.js` 第 229-242 行只校验了时间，未校验"所有记录审查完毕"条件

**位置**: `/TG/tgservice/backend/routes/guest-invitations.js:229-242`

**严重程度**: 🔴 高

**影响**: 
- 管理员可能在助教还未完成约客记录上传时就生成统计，导致数据不准确

---

#### 5. 公休申请审批通过后未校验状态转换合法性

**问题描述**: 
- `applications.js` 第 228-232 行：公休申请审批通过后直接将水牌状态设为"公休"
- 但设计方案 3.2 节规定"任意状态 → 公休审批通过 → 公休"是允许的
- **问题在于**：代码未记录操作日志中的旧值和新值格式不一致（旧值包含 table_no，新值只包含 status）

**位置**: `/TG/tgservice/backend/routes/applications.js:228-232`

**严重程度**: 🟡 中（但影响数据一致性）

---

### 🟡 中等问题（建议修订）

#### 6. 上班 API 未完全遵循状态转换矩阵

**问题描述**: 
- `coaches.js` 第 36-50 行：上班逻辑判断当前状态
- 代码逻辑：如果状态是 `['早加班', '休息', '公休', '请假', '乐捐', '下班']` 之一，则根据班次设置为"早班空闲"或"晚班空闲"
- **问题**：设计方案 3.4 节明确规定"早班空闲/晚班空闲"状态的上班按钮应该灰显（不可用），但代码中只是返回 warning 提示，仍然允许调用

**位置**: `/TG/tgservice/backend/routes/coaches.js:36-50`

**严重程度**: 🟡 中

**建议**: 应该在 API 层面拒绝已在班状态的上班请求，返回 400 错误而非 warning

---

#### 7. 下班 API 状态校验不完整

**问题描述**: 
- `coaches.js` 第 87-94 行：检查是否允许下班的状态列表
- 代码包含：`['早班空闲', '晚班空闲', '早班上桌', '晚班上桌', '早加班', '晚加班']`
- **问题**：设计方案 3.2 节和 3.3 节的下班流程中，"早加班"和"晚加班"状态确实允许下班，但代码未区分"加班后下班"和"正常下班"的场景
- 虽然功能上可行，但缺少操作日志中的备注区分

**位置**: `/TG/tgservice/backend/routes/coaches.js:87-94`

**严重程度**: 🟡 低

---

#### 8. 上下桌单 API 缺少并发控制

**问题描述**: 
- `table-action-orders.js` 第 67-97 行：上下桌单处理逻辑
- 代码先查询水牌状态，然后更新状态，中间存在竞态条件
- 虽然使用了事务，但如果两个请求同时提交，可能导致状态覆盖

**位置**: `/TG/tgservice/backend/routes/table-action-orders.js:67-97`

**严重程度**: 🟡 中

**建议**: 使用数据库锁或乐观锁机制（如检查 updated_at 时间戳）

---

#### 9. 操作日志 old_value/new_value 格式不统一

**问题描述**: 
- 不同 API 中记录操作日志时，old_value 和 new_value 的 JSON 格式不一致
- 例如：
  - `water-boards.js`: `{"status": "...", "table_no": "..."}`
  - `service-orders.js`: `{"status": "..."}`
  - `applications.js`: `{"status": "待处理"}` vs `{"status": "同意"}`（审批时使用文字而非数字）

**位置**: 多个文件

**严重程度**: 🟡 低

**建议**: 统一日志格式规范，参考设计方案 1.8 节的 JSON 序列化规范

---

#### 10. 权限矩阵中"服务员"角色定义缺失

**问题描述**: 
- 设计方案 4.1 节权限矩阵中包含"服务员"角色，所有后台权限均为 ❌
- `permission.js` 中确实定义了"服务员"角色（第 82-91 行）
- **问题**：但设计方案 4.2 节前台权限矩阵中，"服务员"的"内部专用"首页权限为 ❌，但代码中 FRONTEND_PERMISSION_MATRIX 的"服务员"所有功能均为 false，这导致服务员角色实际上无法使用任何功能

**位置**: `/TG/tgservice/backend/middleware/permission.js:82-91`

**严重程度**: 🟡 低（可能是设计如此）

---

**严重程度**: 🟢 低

## 审计总结

天宫国际 V2.0 代码整体实现了设计方案的核心功能，但在以下方面需要改进：

1. **业务逻辑一致性**：约客记录提交路径混乱、状态转换校验不完整
2. **安全性**：存在 SQL 注入风险、硬编码密钥
3. **并发控制**：缺少数据库锁机制
4. **代码规范**：日志格式不统一、缺少输入校验

**建议优先级**：先修复 5 个严重问题，再逐步改进中低优先级问题。
