## 水牌状态测试用例

### TC-006: 水牌状态完整性验证
- 前置条件：
  - 已登录具有水牌管理权限的账号
  - 水牌管理页面已打开
- 测试步骤：
  1. 进入水牌管理页面
  2. 查看状态选择下拉框
  3. 统计所有可选状态数量
- 预期结果：
  - 系统应包含全部 11 种状态：
    1. 早班上桌
    2. 早班空闲
    3. 晚班上桌
    4. 晚班空闲
    5. 早加班
    6. 晚加班
    7. 休息
    8. 公休
    9. 请假
    10. 乐捐
    11. 下班

### TC-007: 水牌状态切换验证
- 前置条件：
  - 已登录具有水牌管理权限的账号
  - 存在至少一个教练水牌
- 测试步骤：
  1. 选择任意教练水牌
  2. 将状态从"下班"切换为"早班上桌"
  3. 保存并刷新页面
  4. 再次切换状态为"晚班空闲"
  5. 保存并刷新页面
- 预期结果：
  - 状态切换成功
  - 刷新后状态保持最新选择
  - 状态变更有记录日志

## 约客管理测试用例

- 前置条件：
  - 已登录具有约客管理权限的账号
  - 当前时间在 16:00 之前
- 测试步骤：
  1. 在 15:59 尝试上传早班约客
  2. 在 16:00 尝试上传早班约客
  3. 在 16:01 尝试上传早班约客
- 预期结果：
  - 15:59 可以成功上传早班约客
  - 16:00 可以成功上传早班约客（含截止时间点）
  - 16:01 上传失败，提示"早班约客上传已截止"

## 异常流程测试用例（新增）

### TC-049: 并发状态变更测试（新增）
- 前置条件：
  - 已登录两个具有水牌管理权限的账号
  - 存在同一个教练水牌
- 测试步骤：
  1. 账号 A 和账号 B 同时尝试修改同一教练的水牌状态
  2. 账号 A 将状态改为"早班上桌"
  3. 账号 B 同时将状态改为"休息"
  4. 查看最终状态和操作日志
- 预期结果：
  - 锁机制生效，只有一个操作成功
  - 另一个操作返回"数据已被修改，请刷新后重试"
  - 操作日志中只记录成功的操作
  - 数据一致性得到保证

### TC-050: 越权访问测试（新增）
- 前置条件：
  - 已创建收银角色账号
  - 已登录收银账号
- 测试步骤：
  1. 使用收银账号尝试访问水牌管理 API（POST /api/water牌/update）
  2. 使用收银账号尝试访问约客管理 API（POST /api/guest/upload）
  3. 使用收银账号尝试访问操作日志 API（GET /api/logs）
- 预期结果：
  - 所有越权访问均返回 403 状态码
  - 返回错误信息："权限不足，无法访问此资源"
  - 操作日志中记录越权访问尝试

### TC-051: 超时提交测试（新增）
- 前置条件：
  - 已登录具有约客管理权限的账号
  - 当前时间为 16:00:01（刚过早班截止时间）
- 测试步骤：
  1. 尝试提交早班约客
  2. 查看系统返回错误
- 预期结果：
  - 提交失败
  - 返回精确错误提示："早班约客上传已截止（截止时间：16:00）"
  - 错误提示中包含当前时间和截止时间

### TC-052: 数据校验失败测试（新增）
- 前置条件：
  - 已登录具有约客管理权限的账号
- 测试步骤：
  1. 提交空数据的约客（客户姓名为空）
  2. 提交非法格式数据（联系方式不是手机号格式）
  3. 提交重复的约客（同一客户同一时间段）
- 预期结果：
  - 空数据提交失败，提示"客户姓名不能为空"
  - 非法格式数据提交失败，提示"联系方式格式不正确"
  - 重复约客提交失败，提示"该客户在此时间段已有预约"
  - 所有错误提示清晰明确
