## 二、发现的问题清单

### 🔴 严重问题（影响测试有效性）

#### 问题 1: TC-201 "自动刷新"与需求"增量添加"矛盾
- **位置**: TC-201 预期结果
- **描述**: 预期结果写"数据每10秒自动刷新一次"，但需求文档明确收银看板使用**增量添加卡片（不刷新整个页面）**。自动刷新和增量添加是两种完全不同的实现方式，此预期结果会误导测试人员验证错误的行为。
- **建议修改**:
  ```
  预期结果应改为：
  - 页面显示自动轮询状态提示（如「监听中·每10秒」）
  - 每10秒检查一次新订单，有新订单时以增量方式追加卡片
  - 无新订单时页面内容不变，不重新渲染
  - 页面无闪烁或白屏
  ```

#### 问题 2: TC-202 警报交互方式与需求不一致
- **位置**: TC-202 操作步骤
- **描述**: 需求文档描述为"页面打开后提醒用户点击开启警报"，即页面首次打开时应有**提醒/提示**。但测试用例直接测试"找到警报开关按钮并点击"，未覆盖"页面打开后的首次提醒"这一关键需求点。
- **建议新增用例**:
  ```
  TC-202b: 页面首次打开时提醒开启警报
  前置条件：首次进入收银看板页面（或 localStorage 中无警报开启记录）
  操作步骤：1. 打开收银看板页面 2. 观察是否有提醒弹窗/提示
  预期结果：- 页面弹出提示（如 Toast 或弹窗）提醒用户开启警报
           - 提示可关闭
           - 关闭后不再重复弹出（记录用户已关闭状态）
  ```

#### 问题 3: TC-404 单选按钮验证标准模糊
- **位置**: TC-404 预期结果
- **描述**: 预期结果写"使用单选按钮（Radio）**或等效的单选交互**"。但需求文档明确强调"**单选按钮（不是下拉框）**"，说明需求方特别关注交互方式，不接受下拉框。测试用例使用模糊描述会失去验证的意义。
- **建议修改**:
  ```
  预期结果应改为：
  - 班次选项仅有「早班」和「晚班」两个
  - 使用单选按钮（Radio Button），不得使用下拉框（Select）
  - 当前班次对应的选项处于选中状态
  - 切换选项时有明显视觉反馈
  ```

#### 问题 4: TC-308 "拖拽修改状态"非需求功能
- **位置**: TC-308 全表
- **描述**: 需求文档描述水牌状态修改为"助教管理可以手动修改移动每个助教的水牌状态，要求易操作"，以及"长按修改状态"。但需求中**并未提及拖拽功能**。TC-308 测试了一个需求文档中不存在的功能，属于超范围测试。
- **建议**: 
  - 选项A：删除 TC-308（需求中不存在拖拽功能）
  - 选项B：如果拖拽是实际实现的功能，应先在需求文档中补充说明

#### 问题 5: TC-601 超出"后台页面Bug修复"范围
- **位置**: TC-601
- **描述**: TC-601 测试"前台登录权限验证"，属于**前台功能**测试。但本测试用例集明确是"V2.0**后台页面**Bug修复验证"。此用例超出了测试范围。
- **建议**: 删除 TC-601 或将其移至独立的"前台功能测试用例集"

- **需求覆盖**: 需求文档 3.3 节「助教约客管理」包含约客审查（判定有效/无效）、约客统计（早班/晚班）、补充上传、统计打开条件（16:00/20:00 + 审查完毕）等后台功能。
- **当前覆盖**: 仅有 TC-503、TC-504 验证菜单项跳转，**无任何功能测试用例**。
- **建议新增用例**:

  TC-701: 早班约客审查 — 判定有效/无效
  TC-702: 早班约客统计 — 生成与展示
  TC-703: 晚班约客审查 — 判定有效/无效

#### 问题 8: TC-203 警报音效预期不完整
- **位置**: TC-203 预期结果
- **描述**: 需求描述"警报每10秒响一次（有新待处理订单时）"。测试用例覆盖了"有待处理订单时每10秒响一次"和"无订单时不响"，但**未覆盖以下场景**：
  - 警报已开启，订单从待处理变为已完成后，警报是否停止？
  - 警报已开启但浏览器被静音时的降级处理？
- **建议补充**:
  ```
  TC-203b: 警报停止条件
  前置条件：警报已开启，有待处理订单，正在播放提示音
  操作步骤：1. 将所有待处理订单处理完成 2. 等待下一个10秒刷新周期
  预期结果：- 待处理订单为0后，提示音停止播放
  
  TC-203c: 浏览器静音时的降级处理
  前置条件：警报已开启，有待处理订单，浏览器已静音
  操作步骤：1. 等待页面刷新 2. 观察视觉反馈
  预期结果：- 无法播放音效时，应有视觉提示（如卡片红色闪烁）作为降级方案
  ```

#### 问题 9: TC-304 "长按修改状态需输入桌号"需求依据不足
- **位置**: TC-304 全表
- **描述**: 需求文档中关于台桌号输入的描述是在「上下桌管理」和「免扫码下单」中，描述为"台桌输入对话框要简单易用，先筛选区域，再用按钮选择台桌"。但需求**并未明确说明通过长按水牌卡片修改为"上桌"状态时需要弹出台桌号输入对话框**。虽然逻辑上合理，但需求中未明确定义此交互。
- **建议**: 明确此交互是否在需求范围内，或在需求文档中补充说明。

#### 问题 10: TC-403 助教卡片信息描述过宽
- **位置**: TC-403 预期结果
- **描述**: 预期结果写"卡片包含班次选择器（单选按钮或下拉框）"。同样使用了"或下拉框"的模糊描述，与需求明确要求的"单选按钮"不符。
- **建议**: 同问题3的修改方式，明确只接受单选按钮。

#### 问题 11: TC-512 收银员权限预期与需求不完全一致
- **位置**: TC-512 预期结果
- **描述**: 预期结果写"收银：仅可见「收银看板」（不应有商品管理等）"。需求文档中收银权限确实只有收银看板，但需求同时说明"其它后台用户，没有后台权限，不能登录后台"。建议补充测试"**无权限用户登录后台时被拒绝**"的场景。
- **建议新增**:
  ```
  TC-512b: 无后台权限用户登录后台
  前置条件：后台用户表中某用户无任何后台权限（非店长/助教管理/前厅管理/收银）
  操作步骤：1. 使用该用户登录后台
  预期结果：- 登录被拒绝，或
           - 登录后侧边栏无任何菜单项，显示空页面
           - 明确提示该账号无后台权限
  ```

## 五、总体评价

### 优点
1. **结构规范**：用例格式统一，前置条件、操作步骤、预期结果完整
2. **核心功能覆盖良好**：用户管理、收银看板、水牌管理、批量更新班次、菜单栏的核心功能点都有对应用例
3. **步骤可执行**：操作步骤描述详尽，测试人员可直接执行
4. **边界场景意识**：有空数据、无修改保存等边界用例（TC-407、TC-408、TC-211）

### 不足
1. **遗漏后台重要功能**：约客审查/统计、申请审批等后台管理功能完全未覆盖（这些也在需求文档中，且属于后台页面功能）
2. **关键描述与需求矛盾**：TC-201 的"自动刷新"与需求的"增量添加"相矛盾
3. **验证标准模糊**：单选按钮验证使用了"或等效交互"的模糊描述
4. **超出范围**：包含前台功能测试用例（TC-601）和需求中未提及的功能（TC-308 拖拽）

### 结论
**NOT PASS** — 测试用例框架良好，但存在需求覆盖不完整、部分用例与需求矛盾的问题。建议按上述 P0 优先级修复后再进入测试执行阶段。
