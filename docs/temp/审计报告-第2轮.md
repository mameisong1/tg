## 新发现问题

### 中优先级问题（未修订，建议补充）

以下问题在第 1 轮审计中标记为"中优先级"，本次修订未涉及，建议评估是否需要补充：

#### 1. coaches 表迁移策略不完整
- **位置**: 8.1 节 数据迁移 SQL
- **问题**: 只提供了 `ALTER TABLE coaches ADD COLUMN shift`，但 SQLite 的 ALTER TABLE 有严格限制：
  - 只能添加列，不能修改/删除列
  - 添加的列必须有 DEFAULT 值或允许 NULL
  - 不能添加约束（如 NOT NULL）
- **建议**: 
  - 方案 A: 明确说明使用 ALTER TABLE 的限制，确保 shift 字段有 DEFAULT 值
  - 方案 B: 如需重建表，提供完整的数据迁移脚本（CREATE TABLE new → INSERT INTO new SELECT → DROP TABLE old → ALTER TABLE new RENAME）

#### 2. 状态转换矩阵 - 加班状态下班流程缺失
- **位置**: 3.2 节 状态转换矩阵
- **问题**: "早加班/晚加班"状态只有→上班按钮→早班空闲/晚班空闲的转换，缺少→下班按钮→下班的转换。实际场景中，助教加班后可能直接下班。
- **建议**: 补充转换规则：
  ```
  早加班 → 下班按钮 → 下班
  晚加班 → 下班按钮 → 下班
  ```

#### 3. 状态转换矩阵 - 上班按钮含义不清晰
- **位置**: 3.2 节 状态转换矩阵 & 3.4 节 按钮可用性矩阵
- **问题**: 3.4 节中"早班空闲/晚班空闲"状态的"上班按钮"为✅，但逻辑上助教已在班，不应再显示"上班"按钮。
- **建议**: 澄清"上班按钮"的真实含义：
  - 如果是指"从非在班状态进入在班状态"，则早班空闲/晚班空闲状态的上班按钮应为❌
  - 如果是指"从加班/休息/请假/乐捐状态恢复为空闲状态"，应重命名按钮为"恢复在岗"

#### 4. 约客审查驳回机制未明确
- **位置**: 2.7.2 节 审查约客记录
- **问题**: 审查结果为"约客无效"后，助教是否可以重新上传约客记录？设计方案未明确。
- **建议**: 补充业务规则：
  - 方案 A: 不允许重新上传，记录为最终无效
  - 方案 B: 允许重新上传，覆盖原记录（需明确覆盖逻辑）

#### 5. 统计生成机制未明确
- **位置**: 2.7.3 节 & 7.1 节
- **问题**: `POST /api/guest-invitations/statistics` 是手动触发还是自动触发？7.1 节提到"审查完毕后开放"，但未说明触发机制。
- **建议**: 明确触发方式：
  - 方案 A: 管理员手动点击"生成统计"按钮
  - 方案 B: 定时任务自动检测审查完毕状态后生成

#### 6. 水牌批量操作 API 缺失
- **位置**: 2.2 节 水牌管理 API
- **问题**: 只有单个助教的水牌状态更新接口，没有批量更新接口。当需要批量设置水牌状态时（如集体下班、批量调整），效率低下。
- **建议**: 新增 `PUT /api/water-boards/batch-status` 接口，支持批量更新多个助教的水牌状态。

### 设计一致性问题（新增）

#### 1. guest_invitation_results 表字段语义重复
- **位置**: 1.7 节
- **问题**: 同时存在 `result` 字段（待审查/未约客/约客有效/约客无效）和 `review_status` 字段（0=待审查，1=已审查），存在语义重复：
  - `result="待审查"` 与 `review_status=0` 表达相同含义
  - `result="约客有效/约客无效/未约客"` 与 `review_status=1` 同时存在
- **建议**: 简化设计，二选一：
  - 方案 A: 保留 `result` 字段，删除 `review_status`，`result` 枚举包含"待审查"
  - 方案 B: 保留 `review_status` 字段表示审查状态，`result` 只保留"未约客/约客有效/约客无效"

#### 2. 应约客统计依赖水牌历史快照，但无历史表
- **位置**: 3.5 节 应约客定义
- **问题**: 统计逻辑注释中提到"需要查询水牌状态历史快照，或使用当时状态记录"，但设计方案中没有水牌状态历史表。`water_boards` 表只有当前状态，无法回溯 16:00/20:00 时点的状态。
- **建议**: 
  - 方案 A: 新增 `water_board_history` 表，记录每次状态变更的时间点和状态值
  - 方案 B: 在 `water_boards` 表中增加 `last_status_change_at` 字段，配合定时任务在 16:00/20:00 快照记录
  - 方案 C: 修改业务规则，统计"当前在班的早班/晚班助教"而非"16:00/20:00 时点的状态"

#### 3. 操作日志 JSON 序列化未说明
- **位置**: 1.8 节 operation_logs 表
- **问题**: `old_value` 和 `new_value` 字段标注为"JSON 格式"，但未说明序列化规则（如字段顺序、null 值处理、日期格式等）。
- **建议**: 补充序列化规范示例：
  ```javascript
  // 示例：水牌状态变更
  old_value: '{"status":"早班空闲","table_no":null}'
  new_value: '{"status":"早班上桌","table_no":"普台 1"}'
  ```
