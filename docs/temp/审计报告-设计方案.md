### 1. 架构可扩展性

**评估：良好**

- ✅ **数据库设计合理**：6 张新增表均有明确的业务语义，主键/外键/唯一约束/索引设计完整，为未来扩展留有余地
- ✅ **RESTful API 设计规范**：接口命名遵循 REST 约定，请求/响应格式统一，支持查询参数筛选
- ✅ **操作日志表（operation_logs）是加分项**：需求文档未要求，但设计主动增加了全链路审计能力，JSON 序列化规范也清晰

**✅ 已完整覆盖的业务逻辑**：

- 水牌状态转换矩阵（11 种状态，含加班 → 下班的修订路径）完整
- 状态按钮可用性矩阵清晰（上班/下班/上桌/下桌/取消）

- 约客驳回后可重新上传覆盖（第 2 轮修订已补充）
- 上下桌单与水牌状态的原子性更新（注意事项中已标注）

**⚠️ 需补充的业务逻辑**：

| # | 问题 | 说明 |
|---|------|------|
| 1 | **收银员完成上下桌单后的水牌状态联动** | 需求文档说"上桌单提交后由收银录入台客多系统"，设计中收银看板有"更新状态"API。但收银员将上桌单标记为"已完成"时，是否需要联动修改水牌状态？当前设计未明确。如果不需要联动（水牌状态由助教的上桌/下桌操作控制），应在设计中明确说明 |
| 2 | **约客统计的"应约客"定义边界** | 需求文档定义"只有在生成统计结果时点，水牌状态是早班空闲/晚班空闲的助教才需要约客"。但设计 3.5 节将应约客状态扩大为 `早班空闲 + 早班上桌`（晚班同理）。**"早班上桌"状态的助教是否应该约客存在歧义**——正在台上的助教可能不方便约客。建议与马哥确认此逻辑 |
| 3 | **服务单/上下桌单的"已完成/已取消"清理机制** | 7.2 节说"定时任务每小时清理一次"，但需求文档说的是"已完成（2小时内），已取消（2小时内）"在收银看板显示。"清理"是指软删除还是仅从看板隐藏？物理删除会丢失操作日志，建议使用软删除或仅过滤不展示 |
| 4 | **助教换班（早班 ↔ 晚班）时水牌状态处理** | 设计文档未说明助教从早班切换为晚班（或反过来）时，如果当前水牌状态是"早班空闲"，是否需要自动调整为"晚班空闲"。建议在批量修改班次 API 中增加状态联动逻辑 |
