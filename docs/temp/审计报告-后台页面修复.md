## 一、coaches.html - 批量修改班次

### 审计结果：❌ NOT PASS

### ISSUE-C1（CRITICAL）：班次枚举值前后端不一致

**问题描述**：
前端 `renderBatchShift()` 提供的班次选项：
```javascript
<option value="早班">早班</option>
<option value="晚班">晚班</option>
<option value="全天">全天</option>    ← 后端不接受
<option value="休息">休息</option>     ← 后端不接受
```

后端 `server.js:2318` 验证逻辑：
```javascript
const validShifts = ['早班', '晚班'];
const finalShift = (shift && validShifts.includes(shift)) ? shift : undefined;
```

**影响**：用户在批量班次界面选择「全天」或「休息」后，保存时后端会将 `finalShift` 设为 `undefined`，然后回退到原有班次值（`currentCoach?.shift`），导致用户操作**静默失效**。用户以为已修改，实际并未生效。

**修复建议**：
- 方案A（推荐）：前后端统一班次枚举，在前端去掉「全天」和「休息」选项
- 方案B：后端扩展 `validShifts` 包含所有合法值

**严重程度**：🔴 CRITICAL - 功能静默失效，用户体验严重受损

### ISSUE-C2（MINOR）：expandAllShifts 函数名与功能不符

**问题描述**：
函数名 `expandAllShifts()` 意为"全部展开"，按钮文字也是"全部展开"，但实际功能是**滚动到第一个已修改的选择框**：
```javascript
function expandAllShifts() {
  const modified = document.querySelectorAll('.batch-shift-select[style*="border-color: rgb(212, 175, 55)"]...');
  if (modified.length > 0) {
    modified[0].scrollIntoView({ behavior: 'smooth', block: 'center' });
  }
}
```

**修复建议**：将函数名和按钮文字改为「定位已修改项」或「滚动到修改处」

### ISSUE-C3（INFO）：批量保存串行调用

**问题描述**：`saveAllShifts()` 使用 `for...of` 逐个调用 `PUT /api/admin/coaches/:coachNo`，没有批量更新 API。助教数量多时等待时间长。

**建议**：后续可考虑增加 `PUT /api/coaches/v2/batch-shift` 批量接口（该接口已存在于 `backend/routes/coaches.js:187`，但前端未使用）

## 三、cashier-dashboard.html - 收银看板三列布局

### 审计结果：❌ NOT PASS

### ISSUE-D1（CRITICAL）：订单完成 API 调用方式与后端不匹配

**问题描述**：

前端代码（第453行）：
```javascript
const endpoint = type === 'service' ? `/api/admin/service-orders/${orderId}` : `/api/admin/orders/${orderId}`;
const res = await api(endpoint, { method: 'PUT', body: JSON.stringify({ status: '已完成' }) });
```

后端实际 API：

| 前端调用 | 后端定义 | 结果 |
|----------|----------|------|
| `PUT /api/admin/orders/:id` | ❌ 不存在 | **404 Not Found** |
| `POST /api/admin/orders/:id/complete` | `server.js:1798` | 正确的端点 |
| `PUT /api/admin/service-orders/:id` | ❌ 不存在 | **404 Not Found** |
| `PUT /api/service-orders/:id/status` | `routes/service-orders.js:157` | 正确的端点 |

**影响**：点击"完成"按钮操作商品订单或服务订单时，**必定失败**（404）。

**修复建议**：
```javascript
// 商品订单：
const endpoint = type === 'service' 
  ? '/api/service-orders/' + orderId + '/status'   // PUT
  : '/api/admin/orders/' + orderId + '/complete';  // POST
const method = type === 'service' ? 'PUT' : 'POST';
```

**严重程度**：🔴 CRITICAL - 核心功能完全不可用

### ISSUE-D2（CRITICAL）：服务订单 API 路径前缀错误

**问题描述**：
- 前端调用：`GET /api/admin/service-orders`
- 后端注册：`app.use('/api/service-orders', ...)`（server.js:299）
- 实际可用路径：`/api/service-orders`（无 `/admin/` 前缀）

**影响**：服务订单列表加载失败，回退到测试数据。虽然不影响页面展示（有 fallback），但不会显示真实数据。

**修复建议**：将 `/api/admin/service-orders` 改为 `/api/service-orders`

### ISSUE-D3（INFO）：助教上下桌单为纯演示模式

**问题描述**：`completeCoachSession()` 仅更新前端内存数据，提示"已确认（演示模式）"，未调用后端 API。

**建议**：后续对接真实的上下桌单 API 时，需确认 `POST /api/table-action-orders` 相关接口

### ISSUE-D4（MINOR）：三列布局缺少响应式适配

**问题描述**：`.three-columns` 使用 `grid-template-columns: repeat(3, 1fr)`，在小屏幕上可能过于拥挤。

### ISSUE-D5（MINOR）：body 设置 overflow:hidden 可能影响移动端

**问题描述**：`body { overflow: hidden }` 会导致某些浏览器无法滚动。建议只在桌面端使用。
