## 代码质量问题

### 问题 1（严重）：后台用户管理 API 缺失 name 字段

**文件**: `backend/server.js`

**位置**:
- 第 1808 行：`SELECT username, role, created_at FROM admin_users` — 缺少 `name`
- 第 1847-1851 行：`INSERT INTO admin_users (username, password, role)` — 缺少 `name`
- 第 1867-1869 行：`UPDATE admin_users SET password = ?, role = ?` — 缺少 `name`

**影响**: 设计方案 1.1 节要求 admin_users 表有 name 字段，API 的增删改查应支持 name。当前 API 无法创建/编辑用户姓名，后台用户管理页面（users.html）的表单也没有姓名输入框。

**修复建议**:
```javascript
// GET: 增加 name 字段
SELECT username, name, role, created_at FROM admin_users

// POST: 增加 name
INSERT INTO admin_users (username, password, name, role) VALUES (?, ?, ?, ?)

// PUT: 增加 name
UPDATE admin_users SET name = ?, role = ? WHERE username = ?
```

### 问题 3（严重）：约客统计生成 SQL 引用未定义变量

**文件**: `backend/routes/guest-invitations.js`

**位置**: POST /statistics handler（约第 250-265 行）

```javascript
// 定义了 placeholders2
const placeholders2 = shouldInviteStatus.map(() => '?').join(',');

// 但 SQL 使用了 placeholders（未在此作用域定义！）
const shouldInviteCoaches = await transaction.all(`
  SELECT wb.coach_no, wb.stage_name, wb.status
  FROM water_boards wb
  INNER JOIN coaches c ON wb.coach_no = c.coach_no
  WHERE wb.status IN (${placeholders})   // ← Bug! 应该是 ${placeholders2}
    AND c.shift = ?
`, [...shouldInviteStatus, shift]);
```

**影响**: 当请求中没有 `date` 查询参数时，`placeholders` 变量不存在（它只在 `if (date)` 块内定义），会导致 `ReferenceError` 崩溃。即使有 date 参数，`${placeholders}` 对应的是 date 的占位符，不是 status 的占位符，SQL 参数绑定也会错位。

**修复**: 将 `${placeholders}` 改为 `${placeholders2}`。

### 偏离设计的地方

1. **权限校验不完整**: 设计方案第 4 节定义了详细的权限矩阵（收银/前厅管理/教练等角色的权限差异），但后端 API 路由注册时只使用了 `auth.required`（登录校验），没有根据角色进行细粒度的权限控制。例如：服务员不应访问水牌管理，但 API 层面未做限制。

2. **收银看板数据聚合**: 设计方案 2.8 节定义了统一的 `/api/cashier-dashboard` API，但实现上拆成了多个独立 API 调用，且商品订单未接入。

3. **admin-login.vue 登录逻辑**: 前端 admin-login.vue 将登录信息存储为 adminInfo，但权限判断依赖角色名称（如 '店长'），而 V2.0 设计中角色值域为中文枚举，需要确保前后端一致。
