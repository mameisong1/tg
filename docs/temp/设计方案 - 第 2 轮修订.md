### 1.7 guest_invitation_results - 约客结果表（新增）

**【第 2 轮修订】** 简化字段设计：删除 review_status 字段，result 枚举包含"待审查"状态。

```sql
CREATE TABLE IF NOT EXISTS guest_invitation_results (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    date TEXT NOT NULL,  -- 日期（YYYY-MM-DD 格式）
    shift TEXT NOT NULL,  -- 早班/晚班
    coach_no TEXT NOT NULL,  -- 助教工号
    stage_name TEXT NOT NULL,  -- 艺名
    invitation_image_url TEXT,  -- 约客记录图片 URL
    result TEXT DEFAULT '待审查',  -- 结果：待审查/未约客/约客有效/约客无效
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    reviewed_at DATETIME,  -- 审查时间
    reviewer_phone TEXT,  -- 审查人手机号
    FOREIGN KEY (coach_no) REFERENCES coaches(coach_no),
    UNIQUE(date, shift, coach_no)  -- 每天每班每个助教只有一条记录
);

CREATE INDEX idx_guest_invitation_date ON guest_invitation_results(date);
CREATE INDEX idx_guest_invitation_shift ON guest_invitation_results(shift);
CREATE INDEX idx_guest_invitation_coach_no ON guest_invitation_results(coach_no);
CREATE INDEX idx_guest_invitation_result ON guest_invitation_results(result);
```

**字段说明**：
- `result` 字段同时表达审查状态和结果：
  - `待审查`：助教已上传约客记录，等待管理员审查
  - `未约客`：助教未上传约客记录（统计时自动标记）
  - `约客有效`：管理员审查通过
  - `约客无效`：管理员审查驳回（**允许助教重新上传覆盖**）

### 1.8 operation_logs - 操作日志表（新增）

新增操作日志表，记录所有状态变更操作。

```sql
CREATE TABLE IF NOT EXISTS operation_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    operator_phone TEXT NOT NULL,  -- 操作人手机号

    target_type TEXT,  -- 目标类型：water_board/guest_invitation/service_order/table_action_order/application
    target_id INTEGER,  -- 目标记录 ID
    old_value TEXT,  -- 旧值（JSON 格式）
    new_value TEXT,  -- 新值（JSON 格式）
    remark TEXT,  -- 备注
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_operation_logs_operator ON operation_logs(operator_phone);
CREATE INDEX idx_operation_logs_type ON operation_logs(operation_type);
CREATE INDEX idx_operation_logs_target ON operation_logs(target_type, target_id);

```

```javascript
// 示例：约客审查
```

### 2.7 约客管理 API

#### 2.7.1 提交约客记录（前端直接上传 OSS）

添加提交时间校验逻辑。

```
POST /api/guest-invitations
Content-Type: application/json

请求体:
{
  "coach_no": "26",
  "date": "2026-04-08",
  "shift": "早班",  // 早班/晚班
  "invitation_image_url": "https://oss.xxx.com/invitation/xxx.jpg"
}

响应:
{
  "success": true,
  "data": { "id": 1, "result": "待审查" }
}

// 时间校验逻辑：
// - 早班约客记录：只能在当天 16:00 前提交
// - 晚班约客记录：只能在当天 20:00 前提交
// - 超时提交返回错误：{ "success": false, "error": "已超过提交截止时间" }
```

#### 2.7.2 审查约客记录

**【第 2 轮修订】** 明确审查驳回后可重新上传。

```
PUT /api/guest-invitations/:id/review
Content-Type: application/json

请求体:
{
  "result": "约客有效",  // 约客有效/约客无效
  "reviewer_phone": "13900139000"  // 审查人手机号
}

响应:
{
  "success": true,
  "data": { 
    "id": 1, 
    "result": "约客有效",
    "reviewed_at": "2026-04-08 16:30:00",
    "reviewer_phone": "13900139000"
  }
}

// 业务规则：
// - 审查结果为"约客无效"后，助教可以重新上传约客记录，覆盖原记录
// - 重新上传后，result 恢复为"待审查"状态，等待再次审查
```

#### 2.7.3 生成约客统计

**【第 2 轮修订】** 按马哥方案修改统计逻辑：首次生成时取当前水牌状态。

```
POST /api/guest-invitations/statistics
Content-Type: application/json

请求体:
{
  "date": "2026-04-08",
  "shift": "早班"  // 早班/晚班
}

响应:
{
  "success": true,
  "data": {
    "date": "2026-04-08",
    "shift": "早班",
    "should_invite_count": 10,  // 应约客人数
    "invited_count": 8,  // 已约客人数
    "invalid_list": [  // 无效约客助教一览
      { "coach_no": "18", "stage_name": "小美" }
    ],
```

### 3.5 应约客定义

**【第 2 轮修订】** 按马哥方案修改统计逻辑：首次生成时取当前水牌状态，不需要历史表。

**应约客定义**：在首次生成约客统计时，水牌状态为以下状态之一的助教：

| 班次 | 统计时点 | 应约客状态 |
|------|----------|------------|
| 早班 | 首次生成早班统计时（通常 16:00 后） | 早班空闲、早班上桌 |
| 晚班 | 首次生成晚班统计时（通常 20:00 后） | 晚班空闲、晚班上桌 |

**特殊情况处理**：

| 统计时状态 | 是否算应约客 | 说明 |
|--------------|--------------|------|
| 早班空闲 | ✅ 是 | 在班且可用 |
| 早班上桌 | ✅ 是 | 在班且正在服务 |
| 早加班 | ❌ 否 | 加班状态，不参与约客 |
| 下班 | ❌ 否 | 已下班 |
| 休息/公休/请假 | ❌ 否 | 不在班 |
| 乐捐 | ❌ 否 | 外出状态 |
| 晚班空闲 | ❌ 否 | 晚班助教，不计入早班统计 |

**统计逻辑**：
```javascript
// 【第 2 轮修订】应约客人数统计 - 首次生成时取当前水牌状态
function countShouldInvite(shift) {
  const shouldInviteStatus = shift === '早班' 
    ? ['早班空闲', '早班上桌'] 
    : ['晚班空闲', '晚班上桌'];
  
  // 直接查询当前 water_boards 表中，状态在 shouldInviteStatus 的助教人数
  // 不需要历史表，用生成统计时的实时状态
  const shouldInviteCount = db.query(
    `SELECT COUNT(*) FROM water_boards 
     WHERE status IN (?) AND coach_no IN (SELECT coach_no FROM coaches WHERE shift = ?)`,
    [shouldInviteStatus, shift]
  );
  
  return shouldInviteCount;
}

// 统计结果生成后缓存，后续查询直接返回缓存结果
// 缓存键：guest_invitation_stats:{date}:{shift}
```

**统计生成机制**：
- 触发方式：管理员手动点击"生成统计"按钮
- 生成条件：
  - 早班：16:00 后 + 所有早班助教约客记录审查完毕
  - 晚班：20:00 后 + 所有晚班助教约客记录审查完毕
- 生成时机：首次生成时快照当前水牌状态，后续查询返回缓存结果

## 9. 注意事项

1. **状态转换原子性**: 水牌状态更新必须与上下桌单创建在同一事务中
2. **权限校验**: 所有 API 必须校验用户权限
3. **OSS 直传**: 前端直接上传 OSS，后端只存储 URL
4. **并发控制**: 水牌状态更新需要加锁，避免并发冲突
5. **数据备份**: 新增表需要加入每日备份脚本
6. **日志记录**: 所有状态变更需要记录操作日志（已通过 operation_logs 表实现）
7. **时间校验**: 约客记录提交需要检查截止时间（早班 16:00 / 晚班 20:00）
8. **约客驳回重传**: 约客审查结果为"约客无效"后，助教可以重新上传覆盖原记录
9. **统计快照**: 约客统计首次生成时快照当前水牌状态，后续查询返回缓存结果
